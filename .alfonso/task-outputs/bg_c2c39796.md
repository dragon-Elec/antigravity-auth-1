- **Issue**: The `RESOLVER_ALIASES` map has `gemini-claude-opus-4-6-thinking-low/medium/high` → `claude-opus-4-6-thinking`, but no entry for `gemini-2.5-pro-low/medium/high` or `gemini-2.5-flash-low/medium/high`. ## Finding 29: `plugin.ts:debugLines` array can grow unbounded
- **Severity**: low
- **Location**: `packages/opencode/src/plugin.ts:1614-1683` and various push sites.
- **Confidence**: medium
- **Issue**: `debugLines` is a `string[]` that receives a push for every event in the request flow (selected, endpoint, model, status, quota, etc.). It’s only consumed at the end as a debug-message block. There is no size cap, so a pathological request flow with many retries can produce a multi-MB debug message. The downstream `injectDebugThinking` then writes that message into the response body as a `thought: true` part, which is then serialized to JSON and sent to the model — wasting tokens and potentially exceeding the model’s context.
- **Evidence**: `plugin.ts:1614-1623`, `request.ts:1814-1819` (`formatDebugLinesForThinking`).
- **Suggested Fix**: Cap `debugLines` at 50 entries (already done in `formatDebugLinesForThinking` which takes `.slice(-50)`, but the array itself is not capped).

## Finding 31: `claude.ts:appendClaudeThinkingHint` line 144 creates a part with text but no type
- **Severity**: low
- **Location**: `packages/core/src/transform/claude.ts:140-152`.
- **Confidence**: medium
- **Issue**: When the existing `systemInstruction` has `parts` but it’s not an array (e.g., a string), the code does `sys.parts = [{ text: hint }]` — but `sys.parts` is set without `sys.role`, so the resulting object is `{ parts: [{ text: hint }] }`. The Gemini API expects `systemInstruction` to be `{ role: "system" (optional), parts: [...] }` and *the parts themselves* may need a `text` field (correct here). However, when the new part is appended to an existing array (line 142), the spread `[...partsValue, { text: hint }]` creates a new array but the part object has no `text` type indicator. The Gemini API has rejected parts with ambiguous types in the past. This is a wire-format concern.
- **Evidence**: `claude.ts:140-152`.
- **Suggested Fix**: Add an explicit `type: "text"` (or the canonical Gemini form) to each system-instruction part.

## Finding 32: `plugin.ts:event` handler accesses `client.session.prompt` without type checking
- **Severity**: low
- **Location**: `packages/opencode/src/plugin.ts:1391-1410`.
- **Confidence**: high
- **Issue**: `client.session.prompt` is called via the `client` reference. If the OpenCode SDK changes the prompt API, this call site will silently fail (it has `.catch(() => {})` to swallow errors). The error is only logged at debug level via `sendIgnoredMessage`. The hidden-prompt injection that the recovery relies on may silently not happen.

## Finding 33: `core/src/agy-transport.ts:serializeRequest` writes body length in hex but skips gzip
- **Severity**: low
- **Location**: `packages/core/src/agy-transport.ts:255-260`.
- **Confidence**: high
- **Issue**: The `Transfer-Encoding: chunked` path serializes the body with `${body.byteLength.toString(16)}\r\n` chunks but only emits one chunk. For SSE streaming, a single chunk is fine, but the chunked format requires a `0\r\n\r\n` terminator (which is present at line 259). The implementation is correct, but it is not `Transfer-Encoding: chunked` compliant in spirit — it’s a single-chunk `Content-Length: 0`-style write. The server may or may not accept this.
- **Evidence**: `agy-transport.ts:255-260`.
- **Suggested Fix**: Either send a single non-chunked body for non-streaming requests and reserve chunked for true streaming, or use proper streaming chunked encoding for the streaming case (splitting on newline boundaries).

## Finding 34: `quota.ts:fetchAvailableModels` projects 403 errors are not retried on the *next* endpoint
- **Severity**: low
- **Location**: `packages/opencode/src/plugin/quota.ts:239-260`.
- **Confidence**: medium
- **Issue**: On 403 with a project id, the code retries the same endpoint without the project. If that also fails with 403, the code falls through to the next endpoint — but if the *next* endpoint also returns 403 with a project, it will retry-without-project *again*, doubling quota usage. Worse, the second endpoint retry does not remember that the first endpoint already 403'd without a project. Should at minimum track per-endpoint retry state.
- **Evidence**: `quota.ts:240-260`.
- **Suggested Fix**: Track per-endpoint 403 retry state and skip endpoints that already 403'd without a project in this fetch cycle.

## Finding 35: `request.ts` calls `parseAntigravityApiBody` twice in some paths

**Overall risk assessment**: The codebase is well-structured and has thoughtful defensive code (signature caching, schema sanitization, thinking-block recovery, multi-account rotation). The most pressing concerns are:

1. The hardcoded OAuth client_secret (Finding 1) — this is a credential leak in published npm packages and should be remediated before any further distribution.
2. The fingerprint identity is weaker than advertised (Finding 2) — operators are led to believe per-account device fingerprints are sent, but only the User-Agent varies, and even that is the agy CLI UA, not the Antigravity desktop UA.
3. The synthetic random project id (Finding 15) silently breaks prompt cache for accounts without a managed project.
4. The third-party version fetch (Finding 8) is a supply-chain risk that can change the User-Agent on every startup.

The remaining findings are quality and robustness issues that should be triaged, but most do not produce immediate user-visible failures.
